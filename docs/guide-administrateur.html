Rôle et accès

L'administrateur gère la partie technique de la plateforme : configuration, comptes utilisateurs, nettoyage de la base de données, monitoring.

Accès disponibles :

Admin Django /admin/ Gestion complète (utilisateurs, pages, documents, paramètres)
Tableau de bord admin /etvous/tableau-de-bord/admin/cleanup/ Nettoyage des sollicitations
Journal d'audit /etvous/tableau-de-bord/journal/ Traçabilité des actions
Inscriptions /etvous/tableau-de-bord/inscriptions/ Validation des comptes (partagé avec le maire)

Middleware

  • ApprovalMiddleware : redirige les utilisateurs non approuvés vers /comptes/pending/. Exempte les URLs d'auth, static, media, home, admin.

Paramètres centralisés (SiteSettings)

Le modèle SiteSettings est un singleton (pk=1) qui centralise toute la configuration du site. Il est mis en cache pendant 5 minutes. Accès : SiteSettings.load() en Python, {{ site_settings.* }} dans les templates.

Paramètres disponibles

GroupeChamps
Identitésite_name, commune_name, population
Coordonnéesaddress, phone, phone_secondary, opening_hours
Email expéditeuremail_from_name, email_from_address
Email contactemail_contact (reçoit les soumissions du formulaire)
SMTPsmtp_host, smtp_port, smtp_username, smtp_password, smtp_use_tls, smtp_use_ssl
Cartographiemap_center_lat/lng, map_default_zoom, mairie_lat/lng
Seuils métierorphan_days (7), cleanup_days (30), stats_period_days (180), reminder_interval_days (7)
Règle absolue : aucune valeur susceptible de changer (coordonnées, téléphone, seuils, GPS...) ne doit être codée en dur dans le code ou les templates. Tout passe par SiteSettings.

Configuration SMTP

Le backend email custom (DatabaseEmailBackend) lit la configuration SMTP depuis SiteSettings. Si aucun hôte SMTP n'est configuré, Django utilise le backend console (affichage terminal, utile en dev).

Pour configurer l'envoi d'emails en production :

  1. Accédez à /admin/settings_app/sitesettings/1/change/
  2. Renseignez smtp_host, smtp_port, smtp_username, smtp_password
  3. Activez smtp_use_tls (recommandé) ou smtp_use_ssl selon votre fournisseur
  4. Configurez email_from_address (l'adresse d'expédition)
  5. Enregistrez

Villages

Les villages sont gérés dans /admin/settings_app/village/. Chaque village a un nom, slug, coordonnées GPS, ordre d'affichage, et un flag is_active. Le champ User.village est une FK vers Village.

Gestion des utilisateurs

Créer un compte élu/secrétaire

  1. Accédez à /admin/accounts/user/add/
  2. Renseignez l'email, le nom, le rôle (mayor, elected, secretary)
  3. Cochez is_approved
  4. Pour la secrétaire : cochez aussi is_staff et attribuez les permissions pages.*
  5. Définissez un mot de passe (obligatoire pour l'accès admin Django)
  6. Enregistrez

Page Équipe municipale

La page « Équipe » affiche automatiquement les utilisateurs ayant le rôle mayor ou elected, triés par function_order. Pour chaque élu, renseignez :

  • function_title : intitulé de la fonction (ex : « Adjoint à l'urbanisme »)
  • function_order : ordre d'affichage (1 = maire, 2 = 1er adjoint, etc.)
  • photo : photo de profil (optionnel)

Changer un rôle

Modifiez le champ role dans l'admin Django. Si vous changez un élu en habitant, ses sollicitations affectées ne sont pas réaffectées automatiquement. Vérifiez manuellement.

Validation des inscriptions

Les inscriptions sont validées via /etvous/tableau-de-bord/inscriptions/ (accessible aux rôles admin et mayor).

Fonctionnement :

  • Un habitant s'inscrit → is_approved=False
  • ApprovalMiddleware le redirige vers /comptes/pending/
  • Notification NEW_REGISTRATION envoyée à tous les admins
  • Admin/maire approuve → email de confirmation envoyé à l'habitant
  • Admin/maire refuse → le compte est supprimé de la BDD

Nettoyage de la base de données

L'interface de nettoyage est accessible à /etvous/tableau-de-bord/admin/cleanup/.

Suppression des sollicitations annulées

Supprime toutes les sollicitations au statut CANCELLED. L'interface affiche le nombre de sollicitations concernées (prévisualisation HTMX en temps réel).

Suppression des sollicitations résolues anciennes

Supprime les sollicitations RESOLVED plus anciennes que N jours (par défaut : cleanup_days = 30, configurable dans SiteSettings).

Actions irréversibles : Les suppressions sont définitives. Les photos et commentaires associés sont supprimés en cascade. Prévoyez un export CSV (fonction maire) avant un nettoyage massif.

Journal d'audit

Le journal est accessible à /etvous/tableau-de-bord/journal/. Il enregistre toutes les actions significatives via le modèle AuditLog.

Actions tracées

ActionCodeDéclencheur
CréationCREATENouvelle sollicitation, nouveau commentaire
Mise à jourUPDATEModification d'une sollicitation
SuppressionDELETENettoyage de sollicitations
Changement de statutSTATUS_CHANGEProgression dans le workflow
AffectationASSIGNPrise en charge ou affectation
ApprobationAPPROVEValidation d'une inscription
ConnexionLOGINConnexion utilisateur (mot de passe ou magic link)

Chaque entrée contient :

  • user : qui a fait l'action
  • action : type d'action
  • target_type, target_id, target_label : l'objet concerné
  • details : métadonnées JSON (ancien statut, nouveau statut, etc.)
  • ip_address : adresse IP du client
  • created_at : horodatage

L'interface supporte le filtrage par type d'action et par utilisateur, avec pagination.

Système de notifications

Architecture

Le service notify() dans apps/notifications/services.py :

  1. Crée une Notification in-app (visible via la cloche)
  2. Vérifie les NotificationPreference de l'utilisateur
  3. Envoie un email HTML+TXT si les préférences l'autorisent et que le SMTP est configuré

Templates email

Les emails sont dans templates/notifications/emails/ : chaque type a un fichier .html (rendu riche) et un .txt (fallback texte brut). Ils héritent de base_email.html qui applique la charte graphique Saulzet.

Relances automatiques

La commande send_reminders envoie des emails de rappel pour les sollicitations orphelines (statut NEW, plus anciennes que le seuil configuré). 

# Prévisualisation (aucun email envoyé)
python manage.py send_reminders --dry-run

# Envoi effectif (seuil par défaut : orphan_days dans SiteSettings)
python manage.py send_reminders

# Seuil personnalisé (3 jours)
python manage.py send_reminders --days 3
Automatisation : Ajoutez cette commande dans un cron quotidien en production : 
# Crontab : relance quotidienne à 8h
0 8 * * * cd /path/to/saulzet-et-vous && /path/to/venv/bin/python manage.py send_reminders

L'intervalle minimal entre deux relances pour la même sollicitation est configurable via SiteSettings.reminder_interval_days (défaut : 7 jours).

Checklist de mise en production

  • Configurer le SMTP dans SiteSettings
  • Migrer la BDD (python manage.py migrate)
  • Compiler le CSS (npm run build:css)
  • Collecter les fichiers statiques (python manage.py collectstatic)

  • Configurer le cron pour send_reminders
  • Mettre à jour le fichier VERSION

Sécurité

Authentification

  • Magic link : token SHA256, expiration 15 min, usage unique
  • Mot de passe : hash Django standard (PBKDF2)
  • Middleware : vérification is_approved sur toutes les requêtes authentifiées
  • Décorateurs : @login_required, @elected_required, @mayor_required, @admin_required

Uploads

  • Validation MIME type (JPG, PNG, HEIC, WEBP uniquement pour les photos)
  • Limite 10 Mo par fichier, 5 photos max par sollicitation
  • Compression automatique (max 1920px largeur)
  • Génération de miniatures

Protection CSRF

Tous les formulaires utilisent le middleware CSRF de Django. Les requêtes HTMX incluent automatiquement le token CSRF.

Traçabilité

Le AuditLog enregistre toutes les actions sensibles avec l'adresse IP, permettant un audit complet en cas de problème.
